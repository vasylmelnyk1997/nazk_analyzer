# Аналіз декларацій НАЗК

### Структура проекту:  
```
nazk_analyzer/  
├── config.py                 // API URLs, налаштування
├── api_client.py             // HTTP запити до НАЗК
├── parser.py                 // маппінг JSON → моделі
├── renderer.py               // моделі → HTML
├── storage.py                // сховище
├── currency_rate_archive.py  // архів курсів валют
└── main.py                   // CLI entry point
```
---
### Організація сховища:  
```
nazk_analyzer/storage/  
├── {doc_id}\_{user_declarant_id}.json             // збережений документ/декларація
├── {user_declarant_id}_{declarant_fullname}.html  // результат аналізу
└── cache/                                         // API responses
    ├── {doc_id}.json                              // список декларацій
    └── {user_declarant_id}.json                   // декларація
```
* Пошук декларації за `doc_id` за маскою `{doc_id}_(*).json`, де (*) - `user_declarant_id`, результатом є json  
* Пошук списку декларацій за `user_declarant_id` за маскою `*_{user_declarant_id}.json`, результатом є масив UUID  
* `declarant_fullname` береться після завантаження хоча б одного документа  
---
### Зовнішній API:  
- налаштовується в окремому файлі  
- root URL:  
   - release ver: https://public-api.nazk.gov.ua/  
   - dev ver: http://127.0.0.1/  
- URI-1 document GET method: */v2/documents/{doc_id}*  
- URI-2 list of documents GET method: */v2/documents/list?user_declarant_id={id}*  
- всі отримані документи зберігати в `storage/cache/` і відповідно спершу шукати в кеші  
- обробка помилок HTTP: видати опис помилки з повідомленням спробувати знову пізніше  
---
### Вхідні параметри (командний рядок, argparse):
- -u: as `user_declarant_id:int` - ідентифікатор декларанта (пріоритет)  
- -d: as `doc_id:UUID` - ідентифікатор декларації НАЗК  
- -r: перезавантажити (ігноруючі cache) відповідно до наданого `id` список (URI-2) + всі документи/декларації цього декларанта з API (URI-1), перегенерувати HTML  
---
### Результат:  
* HTML файл, який буде збережено в сховищі і відкрито в браузері
---
### Алгоритм:  
1. Вказаний `user_declarant_id` параметр  
   1. за наявності `{user_declarant_id}_*.html` і за відсутністю параметру -r, одразу надати готовий html
   2. перевірити наявність списку в сховищі, використовуючи пошук за маскою, за відсутністю зробити запит до URI-2 і
      отримати список декларацій `data[i].id` (as `doc_id`)  
   3. для кожного отриманого `doc_id` перевірити наявність в сховище, за вісутністю
      зробити запит URI-1 (окрім тих, що вже є в сховищі), так само зробити мапінг і зберегти результат до сховища  
   4. сформувати для кожної декларації свою view (html), відповідно до попередніх версій,
      і скомбінувати разом. Загальний дизайн доповнюємо:  
      1. верхня загальна частина з інформацією про декларанта, та склад сім'ї беремо
         за останій відомий поданий рік  
      2. нижня частина з активами:  
         1. зовнішній рівень: роки (вертикальні tabs), сортування зверху-вниз: від більшого до меньшого  
         2. внутрішній рівень: власники (горизонтальні tabs), сортування зліва-направо: 1-декларант, 2-дружина/чоловік, 3-інші  
2. Вказаний `doc_id` параметр  
   1. перевірити наявність документа в сховищі, за відсутністю зробити запит до URI-1  
      1. після кожного API запиту документу виконати мапінг необхідних даних для
         наступного аналізу, відображення і зберігти до сховища  
   2. отримати `user_declarant_id`  
   3. перейти до виконання з пп. 1.1
